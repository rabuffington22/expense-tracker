# Project Intake Wizard

Use this after Ryan gives the basic project direction and before drafting the design plan.

Do not ask every question mechanically. Ask the smallest useful set, infer safe defaults, then ask Ryan to confirm.

## Basic Direction

- What are we building?
- What should exist when the first version is done?
- Is this a software app, game, business workflow, research artifact, document/template, data project, or something else?

## Audience And Use

- Who is this for?
- What will the user do with it first?
- What should the experience feel like?

## Success Criteria

- What would make v1 good enough?
- What must work before we call the first version complete?
- What does not need to work yet?

## Scope Boundaries

- What is explicitly in scope?
- What is explicitly out of scope?
- What should be deferred?
- Are there names, assets, data, systems, people, or references we must avoid?

## Constraints

- Platform or runtime:
- Existing repo, files, tools, or services:
- Data, privacy, PHI, security, or legal constraints:
- Budget, timeline, or deployment constraints:
- Integration constraints:

## Quality Bar

- How polished should v1 be?
- How important are mobile/responsive behavior, accessibility, performance, documentation, tests, and visual design?

## Risks And Unknowns

- What might change the plan?
- What should we prototype or verify early?
- What should trigger a pause for Ryan approval?

## AI Phase Suggestion

After intake, Codex should propose a project-specific phase breakdown.

The proposal should usually separate:

- discovery/design planning;
- build/content work;
- polish/usability;
- testing/verification/handoff.

Adapt the phase names to the project type. Then ask Ryan: "Do you want to confirm this phase breakdown, add anything, remove anything, or rename/reorder phases?"
